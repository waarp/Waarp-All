33  /**
34   * Second CODEC :<br>
35   * - encode/decode : takes a {@link DataBlock} and transforms it to a new {@link
36   * DataBlock} according to the
37   * types<br>
38   * Force ASCII, EBCDIC or IMAGE (with NON PRINT). LOCAL and other subtypes are
39   * not implemented.
40   */
41  @Sharable
42  class FtpDataTypeCodec extends MessageToMessageCodec<DataBlock, DataBlock> {
43    /*
44     * 3.1.1. DATA TYPES Data representations are handled in FTP by a user specifying a representation type. This
45     * type may implicitly (as in ASCII or EBCDIC) or explicitly (as in Local byte) define a byte size for
46     * interpretation which is referred to as the "logical byte size." Note that this has nothing to do with the
47     * byte size used for transmission over the data connection, called the "transfer byte size", and the two
48     * should not be confused. For example, NVT-ASCII has a logical byte size of 8 bits. If the type is Local
49     * byte, then the TYPE command has an obligatory second parameter specifying the logical byte size. The
50     * transfer byte size is always 8 bits. 3.1.1.1. ASCII TYPE This is the default type and must be accepted by
51     * all FTP implementations. It is intended primarily for the transfer of text files, except when both hosts
52     * would find the EBCDIC type more convenient. The sender converts the data from an internal character
53     * representation to the standard 8-bit NVT-ASCII representation (see the Telnet specification). The receiver
54     * will convert the data from the standard form to his own internal form. In accordance with the NVT standard,
55     * the <CRLF> sequence should be used where necessary to denote the end of a line of text. (See the discussion
56     * of file structure at the end of the Section on Data Representation and Storage.) Using the standard
57     * NVT-ASCII representation means that data must be interpreted as 8-bit bytes. The Format parameter for ASCII
58     * and EBCDIC types is discussed below. 3.1.1.2. EBCDIC TYPE This type is intended for efficient transfer
59     * between hosts which use EBCDIC for their internal character representation. For transmission, the data are
60     * represented as 8-bit EBCDIC characters. The character code is the only difference between the functional
61     * specifications of EBCDIC and ASCII types. End-of-line (as opposed to end-of-record--see the discussion of
62     * structure) will probably be rarely used with EBCDIC type for purposes of denoting structure, but where it
63     * is necessary the <NL> character should be used. 3.1.1.3. IMAGE TYPE The data are sent as contiguous bits
64     * which, for transfer, are packed into the 8-bit transfer bytes. The receiving site must store the data as
65     * contiguous bits. The structure of the storage system might necessitate the padding of the file (or of each
66     * record, for a record-structured file) to some convenient boundary (byte, word or block). This padding,
67     * which must be all zeros, may occur only at the end of the file (or at the end of each record) and there
68     * must be a way of identifying the padding bits so that they may be stripped off if the file is retrieved.
69     * The padding transformation should be well publicized to enable a user to process a file at the storage
70     * site. Image type is intended for the efficient storage and retrieval of files and for the transfer of
71     * binary data. It is recommended that this type be accepted by all FTP implementations. 3.1.1.4. LOCAL TYPE
72     * The data is transferred in logical bytes of the size specified by the obligatory second parameter, Byte
73     * size. The value of Byte size must be a decimal integer; there is no default value. The logical byte size is
74     * not necessarily the same as the transfer byte size. If there is a difference in byte sizes, then the
75     * logical bytes should be packed contiguously, disregarding transfer byte boundaries and with any necessary
76     * padding at the end. When the data reaches the receiving host, it will be transformed in a manner dependent
77     * on the logical byte size and the particular host. This transformation must be invertible (i.e., an
78     * identical file can be retrieved if the same parameters are used) and should be well publicized by the FTP
79     * implementors. For example, a user sending 36-bit floating-point numbers to a host with a 32-bit word could
80     * send that data as Local byte with a logical byte size of 36. The receiving host would then be expected to
81     * store the logical bytes so that they could be easily manipulated; in this example putting the 36-bit
82     * logical bytes into 64-bit double words should suffice. In another example, a pair of hosts with a 36-bit
83     * word size may send data to one another in words by using TYPE L 36. The data would be sent in the 8-bit
84     * transmission bytes packed so that 9 transmission bytes carried two host words. 3.1.1.5. FORMAT CONTROL The
85     * types ASCII and EBCDIC also take a second (optional) parameter; this is to indicate what kind of vertical
86     * format control, if any, is associated with a file. The following data representation types are defined in
87     * FTP: A character file may be transferred to a host for one of three purposes: for printing, for storage and
88     * later retrieval, or for processing. If a file is sent for printing, the receiving host must know how the
89     * vertical format control is represented. In the second case, it must be possible to store a file at a host
90     * and then retrieve it later in exactly the same form. Finally, it should be possible to move a file from one
91     * host to another and process the file at the second host without undue trouble. A single ASCII or EBCDIC
92     * format does not satisfy all these conditions. Therefore, these types have a second parameter specifying one
93     * of the following three formats: 3.1.1.5.1. NON PRINT This is the default format to be used if the second
94     * (format) parameter is omitted. Non-print format must be accepted by all FTP implementations. The file need
95     * contain no vertical format information. If it is passed to a printer process, this process may assume
96     * standard values for spacing and margins. Normally, this format will be used with files destined for
97     * processing or just storage. 3.1.1.5.2. TELNET FORMAT CONTROLS The file contains ASCII/EBCDIC vertical
98     * format controls (i.e., <CR>, <LF>, <NL>, <VT>, <FF>) which the printer process will interpret
99     * appropriately. <CRLF>, in exactly this sequence, also denotes end-of-line. 3.1.1.5.2. CARRIAGE CONTROL
100    * (ASA) The file contains ASA (FORTRAN) vertical format control characters. (See RFC 740 Appendix C; and
101    * Communications of the ACM, Vol. 7, No. 10, p. 606, October 1964.) In a line or a record formatted according
102    * to the ASA Standard, the first character is not to be printed. Instead, it should be used to determine the
103    * vertical movement of the paper which should take place before the rest of the record is printed. The ASA
104    * Standard specifies the following control characters: Character Vertical Spacing blank Move paper up one
105    * line 0 Move paper up two lines 1 Move paper to top of next page + No movement, i.e., overprint Clearly
106    * there must be some way for a printer process to distinguish the end of the structural entity. If a file has
107    * record structure (see below) this is no problem; records will be explicitly marked during transfer and
108    * storage. If the file has no record structure, the <CRLF> end-of-line sequence is used to separate printing
109    * lines, but these format effectors are overridden by the ASA controls.
110    */
111   /**
112    * Charset to use
113    */
114   private Charset charsetName;
115 
116   /**
117    * Type of transfer
118    */
119   private TransferType type;
120 
121   /**
122    * Sub Type of transfer
123    */
124   private TransferSubType subType;
125 
126   /**
127    * @param type
128    * @param subType
129    */
130   FtpDataTypeCodec(final TransferType type, final TransferSubType subType) {
131     setCharset(null);
132     this.type = type;
133     this.subType = subType;
134   }
135 
136   /**
137    * @return the subType
138    */
139   public TransferSubType getSubType() {
140     return subType;
141   }
142 
143   /**
144    * @param type the type to set
145    * @param subType the subType to set
146    */
147   public final void setFullType(final TransferType type,
148                                 final TransferSubType subType) {
149     this.type = type;
150     this.subType = subType;
151   }
152 
153   /**
154    * @return the type
155    */
156   public final TransferType getType() {
157     return type;
158   }

160   /**
161    * Set the charset
162    *
163    * @param charset
164    */
165   private void setCharset(final Charset charset) {
166     if (charset == null) {
167       charsetName = Charset.defaultCharset();
168     } else {
169       charsetName = charset;
170     }
171   }
172 
173   @Override
174   protected void decode(final ChannelHandlerContext ctx, final DataBlock msg,
175                         final List<Object> out) throws Exception {
176     // Is an ASCII or EBCDIC mode or IMAGE mode
177     if (type == TransferType.IMAGE) {
178       out.add(msg);
179       return;
180     } else if (type == TransferType.ASCII || type == TransferType.EBCDIC) {
181       final byte[] bytes = msg.getByteBlock();
182       msg.setBlock(decode(bytes));
183       out.add(msg);
184       return;
185     }
186     // Type unimplemented
187     throw new InvalidArgumentException(
188         "Type unimplemented in " + getClass().getName() + " codec " +
189         type.name());
190   }
191 
192   /**
193    * @param byteBuf
194    *
195    * @return the byteBuf
196    */
197   protected final byte[] decode(final byte[] byteBuf) {
198     return new String(byteBuf, type.charset).getBytes(charsetName);
199   }
200 
201   @Override
202   protected void encode(final ChannelHandlerContext ctx, final DataBlock msg,
203                         final List<Object> out) throws Exception {
204     // Is an ASCII or EBCDIC mode or IMAGE mode
205     if (type == TransferType.IMAGE) {
206       out.add(msg);
207       return;
208     } else if (type == TransferType.ASCII || type == TransferType.EBCDIC) {
209       final byte[] bytes = msg.getByteBlock();
210       msg.setBlock(encode(bytes));
211       out.add(msg);
212       return;
213     }
214     // Type unimplemented
215     throw new InvalidArgumentException(
216         "Type unimplemented in " + getClass().getName() + " codec " +
217         type.name());
218   }
219 
220   /**
221    * @param byteBuf
222    *
223    * @return the encoded buffer
224    */
225   protected final byte[] encode(final byte[] byteBuf) {
226     return new String(byteBuf, charsetName).getBytes(type.charset);
227   }
